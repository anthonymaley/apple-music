## Gotchas

- **AppleScript `every track of playlist "X"` is a per-element trap.** `repeat with t in (every track …)` accessing `name of t`/`artist of t` does an Apple Event round-trip per element — ~3.77s for 200 tracks of the 13,646-track library playlist. The bulk form `name of tracks 1 thru n of playlist "X"` (then join in-memory) is ~0.21s (~18×). Always fetch track properties in bulk ranges, never per-element. (Added to `docs/playbook.md`.)
- **`git add X Y` does not scope a commit.** A targeted add followed by `git commit` ships the *entire* staged index. A `git mv` pre-staged by `/kerd:tend` got swept into an unrelated commit this session, shipping a broken README image link for one commit. Run `git diff --cached --stat` before every commit. (Captured via `/shame`; reflection point persisted.)
- **`tracks 1 thru n` errors if n > count.** The bulk fetch clamps n to `count of tracks` first (and guards `n > 0` for empty playlists).

## Insights

- **Measure perf claims before refactoring.** The review hypothesized `onPreview` was a multi-second freeze; timing showed it was 0.41s (it exits at 8). The real offender was `onTracks` (3.77s). Measuring redirected the fix to where it mattered and avoided a wasted change — the doc-fetch/verify discipline applied to performance.
- **A correct fix can create a bug.** The earlier TUI audit de-duplicated *rendering* but left two control loops copy-pasted, which then diverged into the context-mode auto-advance bug (Batch B). Unify control flow, not just leaf rendering.
- **Pure-core + thin-render is the right TUI seam.** Extracting `PlaylistBrowserModel` (zones/badges/duration/enrichment/gradient) made the only unit-testable parts testable (21 tests) while the un-testable rendering stayed thin and was verified live by the user per checkpoint.
